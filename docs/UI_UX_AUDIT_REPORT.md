# Relatório de Diagnóstico de Interface e UX - Lead Finder Zion

| Seção                         | Problema Identificado                                                                                                                                                                                      | Severidade | Recomendação                                                                                                                                  | Impacto                                                                                                                  |
| :---------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :--------- | :-------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------- |
| **1. Consistência de Design** | Uso de cores hardcoded (ex: `bg-[#0066CC]`, `text-[#0066CC]`, `bg-[#F59E0B]`) espalhadas pelos arquivos em vez de usar as variáveis do Tailwind (`bg-primary`, `bg-sidebar`, etc).                         | Alta       | Substituir todas as cores hexadecimais hardcoded por variáveis CSS no `tailwind.config.ts` e `main.css` (ex: `--primary`, `--accent`).        | Facilita a manutenção, suporte a Dark Mode e garante consistência visual em toda a aplicação.                            |
| **1. Consistência de Design** | Espaçamentos e paddings irregulares entre os cards e painéis de filtros nas páginas `Prospeccao`, `Meus Leads` e `InteligenciaZion`.                                                                       | Média      | Padronizar o uso de gaps e paddings (ex: usar `gap-6` e `p-6` globalmente para layouts de grid e cards principais).                           | Melhora a harmonia visual, a hierarquia da informação e o alinhamento da interface.                                      |
| **2. Padrões Shadcn UI**      | Formulários (ex: Login, Alterar Senha, Configurações) não utilizam o componente `<Form>` (react-hook-form + zod) fornecido nativamente pelo Shadcn UI, baseando-se em estados locais simples (`useState`). | Média      | Refatorar formulários complexos para utilizar o ecossistema do Shadcn (`<Form>`, `<FormField>`, `<FormItem>`, `<FormMessage>`).               | Padroniza validações, gerenciamento de estado complexo e melhora a acessibilidade de mensagens de erro.                  |
| **2. Padrões Shadcn UI**      | Uso de `confirm()` nativo do navegador para ações destrutivas (ex: Limpar Cache, Resetar Contador, Excluir).                                                                                               | Alta       | Substituir todas as chamadas nativas de alerta pelo componente `<AlertDialog>` do Shadcn UI.                                                  | Proporciona uma experiência visual muito mais profissional, menos intrusiva e perfeitamente integrada ao tema.           |
| **3. Acessibilidade (WCAG)**  | Botões contendo apenas ícones (ex: botões de ação nas tabelas como Editar, Copiar, Visualizar) carecem de atributos `aria-label` ou texto via `<span className="sr-only">`.                                | Alta       | Adicionar propriedades descritivas a todos os elementos de interface interativos que não possuem texto visível.                               | Permite que usuários com deficiência visual compreendam e operem a aplicação usando leitores de tela sem barreiras.      |
| **3. Acessibilidade (WCAG)**  | Contraste de cor em alguns badges e textos secundários (ex: texto `text-muted-foreground` sobre fundos de mesma matiz) pode não atingir a taxa mínima em alguns monitores.                                 | Média      | Auditar e ajustar as variáveis de cor para garantir o contraste mínimo exigido (ex: 4.5:1 para texto normal conforme WCAG AA).                | Aumenta drasticamente a legibilidade e conforto visual em diferentes monitores e variadas condições de iluminação.       |
| **4. Responsividade**         | Tabelas extensas (como a de `Resultados` na Prospecção) possuem scroll horizontal nativo, porém as ações vitais de paginação não ficam ancoradas e podem se perder em telas móveis.                        | Média      | Fixar os controles de paginação (`sticky bottom`) ou transicionar as tabelas densas para uma visualização em formato de _Cards_ no mobile.    | Melhora de forma exponencial a usabilidade, permitindo navegar pelos resultados com apenas o polegar.                    |
| **4. Responsividade**         | Modais de detalhes com grande volume de dados (ex: `LeadDetailsModal`, `CompareModal`) preenchem toda a largura no mobile dificultando o fechamento ou scroll tátil.                                       | Média      | Ajustar propriedades de largura para garantir áreas de escape laterais, combinadas com scrolls internos e cabeçalhos fixos (`sticky top`).    | Evita frustrações de navegação em smartphones e tablets impedindo cortes na área interativa de visualização.             |
| **5. Padrões de UX**          | Faltam estados elegantes de carregamento (Skeleton loading) ao transitar páginas pesadas (ex: Dashboards e Relatórios), usando atualmente apenas textos simples ou spinners isolados.                      | Média      | Implementar amplamente o componente `<Skeleton>` do Shadcn UI simulando a estrutura da página antes da renderização dos dados.                | Reduz drasticamente a percepção de lentidão (tempo de espera) e previne _Layout Shifts_ (saltos bruscos) do conteúdo.    |
| **5. Padrões de UX**          | Feedbacks visuais de erro retornados da API (notificados via Toasts) por vezes mostram descrições excessivamente técnicas, confusas ou sem tradução direta.                                                | Baixa      | Criar uma camada de interceptação (tratamento de erros) que mapeie códigos técnicos em orientações simples e voltadas ao negócio.             | Elimina a incerteza ou frustração operacional quando cenários de exceção ocorrem no sistema.                             |
| **6. Performance Visual**     | O `App.tsx` realiza o carregamento em lote de todas as páginas simultaneamente, sem divisão de pacotes, elevando a carga inicial desnecessariamente.                                                       | Média      | Implementar `React.lazy()` atrelado ao `<Suspense>` no nível das rotas para particionar dinamicamente o carregamento (Code Splitting).        | Reduz consideravelmente a latência do _Time To Interactive (TTI)_ e o volume de transferência inicial demandado da rede. |
| **6. Performance Visual**     | Animações básicas estão ativas (`animate-fade-in`), mas carecem de micro-interações fluidas (como elevação de sombra, escalas e estados hover aprimorados).                                                | Baixa      | Injetar classes utilitárias de transição ricas (`transition-all duration-300 hover:shadow-lg hover:-translate-y-1`) nos elementos acionáveis. | Converte a navegação em uma experiência muito mais imersiva, moderna e responsiva aos estímulos do cursor/toque.         |
